## 💎 2. 핵심 가치 제안 (Value Proposition)

호텔 매니저가 $200을 결제하는 이유 — **3가지 차별화 강점**:

### 강점 1: 5언어 채널 동시 노출
- 한국어 (호텔이야)
- 영어 (Kotel)
- 일본어 (ホテルだ)
- 중국어 (世界就是家)
- 대만어 / 중국어 번체 (世界就是家)
- 베트남어 (Vietnamese 채널)

→ 다른 마케팅 회사가 따라 하기 어려운 **유일한 자산**

### 강점 2: 6개월 예약 보장 환불
- 1회 $200 결제 → 6개월 동안 예약 0건 시 **100% 환불** (세금 제외)
- 호텔이 안심하고 결제할 수 있는 강력한 명분
- 이걸로 영업 진입 장벽 대폭 낮춤

### 강점 3: 별도 이벤트 페이지 트래픽 송출
- 대표님이 직접 광고비 지출하여 호텔로 트래픽 유입
- 단순 영상 노출이 아닌 **"실제 예약 발생"** 을 위한 추가 마케팅

### 부가 가치
- **3일 내 영상 제작** → 노출 날짜 확정 통보 (이전 약속한 7일 → 3일로 단축)
- **1회 결제 영구 영상 자산** (6개월 후에도 영상은 채널에 남음)

## 💰 3. 가격 정책

| 항목 | 정책 |
|---|---|
| 기본 가격 | **$200 USD (1회 결제)** |
| 보장 기간 | 결제일 기준 **6개월** |
| 결제 수단 | PayPal (Visa/Master/Amex/PayPal 계정) |
| 통화 | USD 단일 (글로벌 영업 위해) |
| 부가세 | PayPal 측 처리 (한국 부가세는 별도 회계 처리) |
| 재계약 가격 | **$200 동일 (할인 없음)** ⭐ |
| 할인/쿠폰 | **운영하지 않음** |

### 재계약 정책 명확화 (2026-04-29 결정)
- ❌ "재계약 시 20% 할인" 같은 정책 **없음**
- ✅ 동일 가격 $200 유지
- 이유: 가격 할인이 아닌 **누적 성과로 설득**
- 메일 문구 예: "지난 6개월 X명 노출 / Y건 예약. 계속 노출을 원하시면 재계약 가능합니다."

## 🔄 4. 환불 정책 (6개월 보장)

### 환불 조건
- 결제일 기준 **6개월 동안 예약 0건** 발생 시
- **100% 환불** (세금 제외)
- 환불 신청은 매니저가 marketing.html에서 직접 가능

### 환불 제외 사항
- 세금 (PayPal 수수료 / 한국 부가세 등)
- 6개월 이전 자발적 해지 (조기 해지는 환불 없음)
- 매니저의 호텔 정보 오류로 인한 노출 실패 (매니저 책임)

### 자동화 트리거
- D-30 (결제 5개월 차): 누적 성과 메일 자동 발송
- D-Day (결제 6개월 차):
  - 예약 0건 시 → "환불 신청 안내" 메일
  - 예약 발생 시 → "재계약 권유" 메일 (할인 없음)

## 🏨 7. 호텔 매칭 정책 (Agoda 매칭)

### 핵심 원칙 (2026-04-26 결정 — 변경 불가)
> **"가입 신청한 호텔은 100% 활성화시킨다."**
>
> 매칭 = 매출의 시작점. 매칭 실패 = 매출 0.
> 단 한 명의 매니저도 놓치지 않는다.

### 3단계 안전망 (이미 구현 완료, 라이브 작동 중)

```
[1차] 자동 매칭 (Google Places + Agoda 동시 검색)
  ✅ 성공 (예상 70~80%) → 즉시 활성화
  ❌ 실패 → 2차로

[2차] 매니저 수동 입력 (Agoda URL 백업)
  - "호텔을 못 찾으셨나요? 아고다 URL로 입력하기" 백업 필드
  ✅ 성공 (예상 10~15% 추가) → 활성화
  ❌ 실패 → 3차로

[3차] 관리자 수동 매칭 (admin.html → Agoda Matching 탭)
  ✅ 옵션 ①: hotelId 직접 입력 → 활성화
  ✅ 옵션 ②: Agoda 등록 안내 메일 자동 발송
  ✅ 옵션 ③: 1:1 컨택 (대표님이 직접 매니저에게 연락)
```

### 호텔 등급 정책
- **3성급 이상만 등록 가능** (Agoda 자동 등급 표시)
- 1~2성급 / 게스트하우스는 영상 품질 유지 위해 제외
- APA 호텔은 전면 제외 (대표님 정책)

## 🔑 7-A. 매니저 정보 변경 정책 (3-Tier 차등) ⭐ 2026-04-29

### 핵심 원칙
> **매니저 셀프 서비스 + 비즈니스 영향도에 따라 승인 절차 차등**

### Tier 1: 즉시 변경 (승인 불필요)
**매니저 본인 정보만 영향**
| 항목 | 처리 |
|---|---|
| 비밀번호 | 즉시 변경 (현재 settings.html 작동 중) |
| 휴대폰 | 즉시 변경 + 이력 로그 |
| 직책 | 즉시 변경 + 이력 로그 |
| 마케팅 수신 동의 | 즉시 변경 |

### Tier 2: 즉시 변경 + 관리자 알림
**비즈니스에 약간 영향, 검증 필요할 수도 있음**
| 항목 | 처리 |
|---|---|
| 이메일 변경 | 새 이메일 재인증 + 관리자 알림 |
| 호텔 사진 추가/변경 | 즉시 반영 + 관리자 알림 (부적절 시 사후 제거) |
| 담당자 이름 오타 수정 | 즉시 반영 + 알림 |

### Tier 3: 관리자 승인 필수 ⭐
**비즈니스 핵심에 영향, 사전 검증 필요**
| 항목 | 처리 |
|---|---|
| 호텔명 변경 | 매니저 신청 → 관리자 승인 → 반영 |
| 주소 변경 | 매니저 신청 → 관리자 승인 → 반영 |
| Agoda 링크 변경 | 매니저 신청 → 관리자 검증 → 반영 |
| 호텔 등급 변경 | 매니저 신청 → 관리자 검증 → 반영 |
| 호텔 담당자 교체 | 인수인계 워크플로 (별도) |

### 왜 Tier 3 승인 필수인가
- **호텔 바꿔치기 방지**: 결제 후 다른 호텔로 변경 시도 차단
- **영상 재제작 비용 방지**: 잘못된 정보로 영상 재작업 발생 방지
- **6개월 보장 약속 보호**: 우리는 호텔에 약속한 입장 → 신중한 변경

### 구현
- settings.html에서 Tier 1, 2 셀프 처리
- "호텔 정보 변경 신청" 버튼 → Tier 3 승인 큐로
- admin.html에 "변경 승인 대기" 탭 신설 (BACKLOG)

## 👥 7-B. 호텔 담당자 교체 정책 (Manager Handover) ⭐ 2026-04-29

### 핵심 원칙
> **"매니저는 떠나도 호텔은 남는다. 결제 이력은 호텔에 영구 귀속."**

### 데이터 모델

```
hotels (호텔)
  └─ hotel_managers (담당자 이력) ⭐ 새 테이블
       ├─ user_id        (매니저 ID)
       ├─ hotel_id       (호텔 ID)
       ├─ role           (primary / secondary)
       ├─ current        (현재 담당 여부)
       ├─ started_at     (담당 시작일)
       ├─ ended_at       (담당 종료일, NULL이면 현재)
       └─ handover_from  (인수인계 받은 사람 ID)

payments (결제)
  └─ paid_by_user_id (결제 당시 매니저 ID, 영구 보존)
```

### 인수인계 워크플로

```
[Step 1] 김매니저: settings.html → "호텔 담당 인수인계"
[Step 2] 새 담당자 이메일 입력 → 박매니저
[Step 3] 박매니저에게 초대 메일 발송 (인수인계 토큰 포함)
[Step 4] 박매니저 메일 클릭 → 수락
  ├─ 신규자: 가입 → 자동으로 hotel_managers 추가
  └─ 기존자: 호텔 담당자로 추가
[Step 5] 시스템:
  - 김매니저: current=false, ended_at=now, role=former
  - 박매니저: current=true, role=primary
  - 결제 이력 그대로 보존
[Step 6] 관리자 ops 메일 알림
```

### 권한 매트릭스

| 권한 | primary (현재) | secondary (현재) | former (이전) |
|---|---|---|---|
| 호텔 정보 조회 | ✅ | ✅ | ✅ (read-only) |
| 호텔 정보 변경 신청 | ✅ | 🔒 (primary 동의 필요) | ❌ |
| 결제 신청 | ✅ | ❌ | ❌ |
| 마케팅 페이지 조회 | ✅ | ✅ | ✅ |
| PDF 보고서 다운로드 | ✅ | ✅ | ✅ |
| 담당자 교체 신청 | ✅ | ❌ | ❌ |

### 여러 담당자 가능
- **primary 1명** + **secondary 여러 명**
- 마케팅 팀장 (primary) + 마케팅 직원 (secondary)
- secondary는 정보 변경 권한 제한

### 이전 담당자 권한
- 본인이 결제 참여한 호텔 정보 평생 조회 가능
- 변경 권한은 없음
- 새 결제 시작 시 다시 primary 가능

### 구현
- 새 테이블: `hotel_managers`
- API 신설: `/api/hotel-handover.js` (초대/수락/거절)
- UI: settings.html 인수인계 섹션 + 초대 수락 페이지
- ops 메일: 인수인계 발생 시 자동 알림

**기술 구현**:

## 📊 10. 매니저 페이지 우선순위 (marketing.html)

### 핵심 원칙 (2026-04-29 정정) ⭐
> **"조회수가 가짜 지표 (Vanity Metric), 예약이 진짜 지표 (Real Metric)"**
>
> 영상 조회수가 낮을 수도 있다. 그러나 예약이 발생하면 비즈니스 가치 100%.
> 매니저에게 "조회수보다 예약이 진짜 가치다" 라고 학습시킨다.

### "매니저가 페이지 들어왔을 때 3초 안에 봐야 할 것" 우선순위

```
🥇 1순위 (가장 크게) ⭐ 진짜 가치
   "📞 8건의 예약이 발생했습니다"
   "💰 추정 매출: $12,800"
   "🎯 ROI: 64배 (1$ 투자 → 64$ 매출)"

🥈 2순위 (예약 검증 — 신뢰의 핵심)
   "[예약 상세 보기] 클릭"
   - 예약번호 / 날짜 / 금액 / 국가
   - "Agoda Partner Hub에서 직접 확인하세요" 링크
   ← 매니저가 본인 눈으로 검증 가능 (BUSINESS.md §7-D 참조)

🥉 3순위 (참고 지표)
   📺 채널별 조회수 (작게)
   - 한국어 / 영어 / 일본어 / 대만어 / 베트남어
   ← 5개 채널 노출 가치는 증명하되, 메인 지표는 아님

🔻 4순위 (서브 정보)
   - D-Day 카운트 (6개월 보장 남은 일수)
   - 호텔 스토리 (우리와 함께한 시간)
   - 다음 리포트 발송일

🔻 5순위 (한 번 보면 끝)
   - 호텔 정보 (수정 신청 링크)
   - 결제 내역 (날짜+시간)
   - 인보이스 다운로드
```

### 왜 예약을 최우선에 두는가 (대표님 통찰)
- **조회수 5,000 / 예약 0건** → 매니저 시각: "광고 효과 없음"
- **조회수 500 / 예약 12건** → 매니저 시각: "와우! 이게 진짜 가치"
- 조회수는 **변동성 큼** (영상 알고리즘, 계절, 트렌드)
- 예약은 **실제 매출 발생** (호텔 관점에서 진짜 ROI)

### 핵심 인사이트
- **dashboard 개념 폐기 → "예약 검증 페이지" 개념으로 재정의**
- 매니저는 매일 들어오지 않음
- 들어올 때마다 **"실제 예약이 발생하고 있다 + 직접 검증 가능"** 확신
- → 신뢰 → 재계약 → LTV 증가
