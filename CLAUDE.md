# ⚠️ 최우선 지시 — 작업 시작 전 필수

**모든 새 채팅 / 새 세션의 Claude는 작업 시작 전 반드시 다음을 수행한다:**

## 1단계 — 헌법 읽기 (필수)
GitHub repo의 [`OPERATIONS_CHARTER.md`](./OPERATIONS_CHARTER.md)를 가장 먼저 읽고, 11대 원칙을 확인한다.

## 2단계 — 자가 검증 의무
어떤 제안을 드리기 전에, 헌법의 11대 원칙 자가 검증을 통과해야 한다. 통과 못 하는 제안은 절대 드리지 않는다.

## 3단계 — "헌법 확인" 명령어 인식
대표님이 **"헌법 확인"** 이라고 한 마디 하시면, Claude는 즉시 작업을 중단하고, 헌법을 다시 읽고, 어떤 원칙 위반인지 자가 진단한 후 다른 길을 찾는다.

## 4단계 — 금지 표현
다음 표현은 헌법 위반 신호이며, Claude의 입에서 나오면 안 된다:
- ❌ "이건 푸시가 안 되니까 다른 방법으로..."
- ❌ "로컬에서 먼저 테스트하시죠..."
- ❌ "임시로 이렇게..."
- ❌ "PC에서 PowerShell 열고..."
- ❌ "외근모드 명령어 붙여넣기..."

올바른 행동은 **막힌 사실만 보고드리고 멈추는 것**이다. 우회 제안은 대표님 명시적 허락 없이는 절대 금지.

## 3. 절대 지킬 8가지 (NEVER 룰)

### ① admin-business / admin-gallery 무변경 원칙
- `admin-gallery.html` 은 **5대 핵심 참조 문서 중 하나**.
- 대표님 명시 지시 없이 디자인/구조 변경 금지.
- 수정 필요 시 반드시 사전에 보고 후 승인 받기.

### ② 백업 자동 (_backup_YYYYMMDD)
- HTML 파일 수정 전 **반드시** 동일 디렉토리에 `파일명_backup_YYYYMMDD.html` 생성.
- 예: `manager-dashboard.html` → `manager-dashboard_backup_20260503.html`
- `.gitignore`에 `_backup_*/` 등록되어 있어 커밋되지 않음 (정상).

### ③ CHANGELOG.md `[변경사유]` 태그 필수
- 모든 코드 수정은 `CHANGELOG.md` 에 기록.
- 형식:
  ```
  ## YYYY-MM-DD
  - [변경사유: 사유 요약] 변경 내용
  ```
- git commit 메시지에도 `[변경사유]` 태그 포함:
  ```
  fix: 모달 닫기 버튼 null-safe 처리 [변경사유: 콘솔 에러 P0]
  ```

### ③-2 작업 종결 신호 — auto-detect-bot용 ★ 2026-05-06 추가
- 작업 ID(BL-XXX 등)가 포함된 commit에는 **반드시** 다음 둘 중 하나의 신호가 명시되어야 한다:
  - `[start]` 또는 `[wip]` — 작업 진행 중 (auto-bot이 status: in_progress 마킹)
  - `[done]` 또는 `완료` 또는 `✅` — 작업 완료 (auto-bot이 status: done + completed_at 마킹)
- **마지막 보고 직전, Claude는 반드시 종결 commit을 푸시한다:**
  ```
  chore(BL-XXX): 작업 완료 [done] [변경사유: 그림 일치 OS 1단계 종료]
  ```
- 신호가 없으면 auto-bot이 fix/feat 키워드로 in_progress로 잘못 추론하므로
  **작업 끝났는데 진행 중으로 박히는 모순 발생** (2026-05-06 BL-LIVE-STATUS-PANEL 사례).
- 한 commit에 여러 작업 ID가 들어가는 경우 모두 같은 신호로 처리됨.
  다른 신호 필요 시 작업별 별도 commit 권장.

### 11.3 사람용 탭 자동 추출기와의 약속

`admin-status.html`의 `loadHumanTab` 함수는 다음 규칙으로 동작한다:

1. **5블록 헤딩(`## 🎯`, `## 📍`, `## 🛠`, `## ✅`, `## ⏱`) 추출 시도**
2. **추출 성공 (5블록 모두 발견)** → 사업가 카드 UI로 표시 + 기술 상세는 `<details>` 접기
3. **추출 실패 (5블록 없음 — 표준 박기 전 chat-log)** → 자동 용어 변환 규칙(11.2 표) 적용 후 표시 + 안내 배지

이 약속은 양방향이다:
- **chat-log 작성 시 5블록 누락하면 자동 변환에 의존하게 됨 → 변환 정확도 한계로 인해 어색할 수 있음**
- **5블록을 정석대로 박으면 카드 UI로 깔끔하게 표시됨**

### 11.4 강제 규칙 — 다시 묻지 않기

- 새 chat-log 박을 때마다 표준 적용 여부 대표님께 묻지 않는다 (이 표준이 답이다)
- 표준 위반 chat-log는 사람용 탭에서 ⚠️ 배지로 자동 표시 (대표님께 자체 보고)
- 표준은 헌법 6조 + 헌법 12조 자체 검증 의무에 종속되며, 변경 시에만 대표님 승인 필요

## 12. 채팅 라우팅 판단 의무 ★ 2026-05-06 박힘

> **목적:** 작업 도중 채팅창이 닫혀서 작업이 중간에 끊기는 사고 방지.
>
> **배경:** 한 채팅창이 너무 길어지면 시스템이 강제 종료시키거나 응답이 도중에 잘릴 수 있다. 이 사고가 작업 진행 중에 발생하면 어디까지 됐는지 잃어버리고 시스템이 깨진다 (예: commit은 했는데 push 못 함). 그래서 **새 작업을 시작하기 전에 "이 채팅에서 끝까지 갈 수 있나"를 Claude가 자체 판단**하고 대표님께 알려야 한다.

### 12.1 의무 — 인계서 받으면 첫 줄에 라우팅 판단 명시

대표님이 admin-status.html의 카드 클릭으로 받은 인계서를 채팅창에 붙여넣었을 때, Claude는 **응답의 가장 첫 줄**에 다음 형식으로 라우팅 판단을 명시한다.

```
🚦 채팅 라우팅 판단:
[✅ 이 채팅 진행 가능 | ⚠️ 새 채팅 권장 | 🚨 새 채팅 강제]

사유:
- (사유 1)
- (사유 2)
```

이 표시는 **선택이 아니라 의무**다. 인계서를 받았는데 라우팅 판단 없이 작업을 바로 시작하면 헌법 12조 위반이다.

### 12.2 판단 기준 — 3단계 신호

#### 🚨 새 채팅 강제 (즉시 멈춤, 작업 시작 안 함)
다음 중 하나라도 해당하면 즉시 멈추고 "새 채팅 열어주세요"만 안내:
- 직전 응답 도중 토큰 한계 메시지가 있었음
- 직전 응답 도중 도구 호출 실패가 반복됨 (3회 이상)
- 대표님이 "끊김" / "느려짐" / "도중에 멈춤" / "응답이 짧아짐" 등 신호를 직접 보냄
- 이 채팅에서 이미 8개 이상 작업 완료 + 다음 작업이 large

#### ⚠️ 새 채팅 권장 (대표님께 한 번 확인)
다음 중 하나라도 해당하면 "진행할까요? 또는 새 채팅?" 한 번 묻기:
- 이 채팅에서 이미 4개 이상 작업 완료
- 직전 응답이 매우 길었음 (도구 호출 15회 이상 또는 응답 길이 매우 김)
- 다음 작업이 large + 예상 시간 ≥ 3h
- 직전 응답에서 시스템 경고/오류가 있었음
- 같은 채팅에서 이미 한 번 "끊김" 신호가 있었음

#### ✅ 이 채팅 진행 가능 (자동 진행)
위 두 조건에 해당하지 않으면 그대로 작업 시작.

### 12.3 강제 새 채팅 시 응답 형식

```
🚨 새 채팅 강제

사유:
- (구체 사유)

작업을 시작하지 않고 멈춥니다. 다음 단계:
1. 새 채팅창을 여세요
2. 같은 인계서를 새 채팅에 붙여넣으세요
3. 거기서부터 작업이 자동 진행됩니다

이 채팅창은 그대로 두셔도 됩니다 (참고용).
```

### 12.4 권장 시 응답 형식

```
⚠️ 새 채팅 권장

사유:
- (구체 사유)

진행 방향 결정 부탁:
① 이 채팅에서 그대로 진행 (위험 감수)
② 새 채팅으로 이동 (안전)

"진행" 또는 "새 채팅" 한 마디 주시면 그대로 가겠습니다.
```

### 12.5 진행 가능 시 응답 형식

```
✅ 이 채팅 진행 가능

(곧바로 작업 시작 — 별도 확인 묻지 않음)
```

### 12.6 작업 도중 신호 감지 시 — 자체 끊기 의무

작업이 시작된 후에도 다음 신호가 보이면 Claude는 **자체적으로 작업을 끊고 안전 지점에 멈춤**:

- 응답이 점점 짧아짐 (토큰 압박 신호)
- 도구 호출이 실패하기 시작함
- 응답 도중 중단됐다가 재개됨

**자체 끊기 시 행동:**
1. 현재까지 완료한 부분 git commit + push (안전 지점 확보)
2. 대표님께 보고: "여기까지 끝냈습니다. 새 채팅 권장합니다."
3. 다음 작업 인계서 미리 클립보드 형식으로 제공

### 12.7 카드 클릭 인계서가 아닌 일반 대화에도 적용

대표님이 카드 클릭이 아니라 일반 메시지로 작업을 지시한 경우에도, 다음 신호가 있으면 같은 라우팅 판단을 첫 줄에 명시:

- 이 채팅에서 이미 4개 이상 작업 완료한 상태에서 새 작업 지시
- 일반 메시지에 "BL-XXX 진행해" 같은 작업 시작 지시가 명시됨
- 분량이 큰 새 작업이 시작되는 시점

### 12.8 거짓말 금지

라우팅 판단은 **Claude가 직접 자체 점검한 결과**여야 한다. 다음은 금지:

- "잘 모르겠지만 진행 가능"이라고 추측하는 것 (모르면 ⚠️ 권장으로)
- 작업하고 싶어서 "진행 가능"으로 우기는 것
- 같은 채팅 길이인데 매번 다른 판단 내리는 것 (일관성)

판단 근거를 사유 필드에 구체적으로 적는다. 예: "이 채팅에서 4개 작업 완료(BL-COMMAND-CENTER, BL-STATUS-REORDER, ...)" 같이 셀 수 있는 사실로.
