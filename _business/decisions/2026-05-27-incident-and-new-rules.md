## D12. base64 절대 금지, plain text only ⚠️ 최고 우선순위

**한 줄:** 자동 창구(`/api/ops/github-commit`)에 보내는 `content`는 무조건 plain text.

**사고 경위 (오늘 오전):**
- Claude가 큰 파일 보낼 때 base64로 인코딩 → 자동 창구가 base64 문자열 그대로 파일에 박음 → 라이브 4개 깨짐
- 복구 commit 4개로 원복: `dd9d4528`, `3836a614`, `c4dcfd63`, `54f8c2d7`

**룰:**
```
✅ 올바름:
{
  "path": "_admin/admin.html",
  "content": "<!DOCTYPE html>\n<html>...",   ← 원본 텍스트 그대로
  "message": "fix: ..."
}

❌ 금지:
{
  "path": "_admin/admin.html",
  "content": "PCFET0NUWVBFIGh0bWw+...",        ← base64 인코딩 절대 금지
  "message": "fix: ..."
}
```

→ **결과:** 어떤 큰 파일도 plain text로 박힘. base64 사고 영구 차단.

## D13. 매 commit 직후 라이브 검증 의무

**한 줄:** "200 응답" = 성공 아님. 라이브 URL 직접 확인이 진짜 성공.

**검증 순서:**
1. `/api/ops/github-commit` 호출 → 200 받음
2. **즉시** `curl https://raw.githubusercontent.com/.../path` 로 raw 확인
3. 15자 이상 핵심 문자열 grep 일치 확인
4. 라이브 URL `https://gohotelwinners.com/...` 으로 최종 확인

**대표님께 보고할 때:** "commit hash X" 아님. **"어디 가서 뭐 누르면 뭐 보임"** + 라이브 스크린샷.

→ **결과:** 사고 후 한참 지나 발견하는 일 0건.

## D15. 활동 이력 = 채팅 인계 백업 단일 진실원 🆕 오늘 새 사실

**한 줄:** GitHub 인계 메모가 실패해도 활동 이력에서 복구 가능.

**오늘 증명된 사실:**
- 이전 채팅이 GitHub 인계 메모만 박고 채팅 텍스트 박스 안 줌
- 새 채팅 Claude의 도구 환경이 달라서 GitHub raw fetch 실패
- → 멈춤 사고 발생
- **해결: 활동 이력(`recent_chats`) 검색 → 어제부터 오늘까지 5개 채팅 흐름 다 파악 → B/C/A 작업 정의 복원 성공**

**룰:**
1. 채팅 종료 시 **GitHub commit + 채팅 텍스트 박스 둘 다** 제공 (D10 ④번)
2. **둘 다 실패해도** 새 채팅에서 `recent_chats` 호출하면 복구 가능
3. 그래서 **모든 결정/사고/룰을 채팅에 명시적으로 작성**해야 활동 이력에 남음 (마음속 결정 ❌)

→ **결과:** 어떤 인계 실패가 일어나도 활동 이력만 살아있으면 100% 복구.
