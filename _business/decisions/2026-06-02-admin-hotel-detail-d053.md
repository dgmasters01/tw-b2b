## 배경

매니저 허브(admin-manager-hub)는 요약 KPI만 보여줘서, 대표님이 호텔별로 예약·통계를 보며 상담하기엔 부족했다. 대표님 원칙: **매니저 페이지에 이미 만든 형태(예약 세부·통계·회차·분석 전체)를 admin으로 가져와 동기화**한다 — 새 레이아웃 발명 금지.

## 확정 내용

1. admin 호텔 상세 = 매니저 분석 형태 미러링, **별도 페이지**로 분리(허브 목록 → 호텔 클릭 → 상세). UX/UI는 클로드 판단 위임받음 → 별도 페이지 선택(넓은 공간 + URL 단위 + 매니저 분석이 독립 풀페이지라 형태 일치).
2. 탭 = 개요·채널별·패턴·예약 상세. 비교용 나라/도시/호텔/성급 탭은 단일 호텔에 안 맞아 제외.
3. 예약 컬럼 = 매니저 페이지 그대로(예약번호·예약일·체크인·체크아웃·금액·수수료·상태·기기).
4. **수수료 2버전**: 매니저 화면 비노출(우리 아고다 제휴수익) / admin 화면 노출. 같은 컴포넌트 + 표시 플래그.
5. **기간 4구분**: 마케팅 전 / 마케팅 기간 / 마케팅 후 / 전체. "전체"는 분석 메뉴와 같은 소스로 동기화(숫자 불일치 방지).
6. 회차 = campaign_log 기반. 기간 구분은 회차별 6개월 보장기간으로 예약 자동 분류.
7. 마케팅 전 예약 = 과거 아고다 데이터(booking-analytics 인라인)에서 호텔명+도시+국가 매칭 → 대표님 확정 게이트(오매칭 방지). 상담 스토리: "유료 전에도 이만큼, 유료 기간엔 이만큼."
8. 담당 매니저 이름·연락처 = 가입 폼 추가 + 상세 페이지 표시.

## 근거 (코드 확인)

- booking-analytics.html = is_admin 전용 전체 분석(매니저용 아님).
- dashboard.html = 수수료 미표시 → 매니저 비노출 관행 확인.
- signup.html = 매니저 이름/연락처 필드 부재.
- campaign_log / cycle_stage / 보장기간 = 이미 존재.

## 미정

- "마케팅 전 예약" 명칭(추천=마케팅 전 예약).
- admin.html members 데이터 구조 확인.
- admin 토큰의 임의 호텔 예약 조회 권한(RLS) 검증.
