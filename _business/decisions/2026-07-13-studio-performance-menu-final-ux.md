# 스튜디오 성과표 메뉴 — 최종 UI/UX 확정 + 서비스 방향 (2026-07-13)

> **성격:** D-063(성과표=전 채널·영상·호텔 종합 조망)의 **구현 레벨 최종 확정**.
> 이 문서는 "무엇을 · 어떻게 보여주고 · **왜** 그렇게 보여주는지"(서비스 방향)를 남긴다.
> 결정자: 대표님 / 정리: 클로드. 이후 딴 채팅에서 이어갈 때 이 문서가 근거다.

---

## 0. 이 메뉴의 목적 (왜 존재하나)

성과표 = **"일·주·월 성과를 채널별로 5초 안에 보고 → 다음 행동을 정하는" 판.**
숫자만 보는 판이 아니다. 핵심은 **판단 → 행동**으로 잇는 것:
- 잘되는 채널·호텔 = **더 민다** (콘텐츠를 그쪽으로).
- 새는(취소 많은) 채널·호텔 = 눌러서 **호텔 메뉴 ⑤(취소·확정률 전략)로 고치러 간다.**

근거: 예전 매출 대시보드 때 정한 북극성("일·주·월 매출을 채널별로 5초 안에 한 화면") + 오늘 대표님 지시("어떻게 보여주고 **왜** 보여주는지가 서비스 방향"). 이 둘을 성과표에 그대로 박았다.

**경계:** 성과표=전체 조망 / 호텔 메뉴(D-062)=호텔 1개 심층. 성과표에서 문제를 찾고 → 호텔 상세로 드릴다운.

## 1. 예약금액 공개 원칙 (중요 — 오늘 재확정, 기존 원칙 갱신)

**결정: 예약금액은 에디터 포함 전원 공개. 수수료(우리 커미션)만 대표님(owner) 전용.**

- **예약금액** = 손님이 호텔에 낸 돈 = **호텔 매출**. 우리 돈 아님. "우리가 이 호텔에 이만큼 예약을 만들어줬다"는 **우리 서비스 성적표**. 화면 라벨은 오해 방지 위해 `예약금액(호텔이 받은 돈)`.
- **수수료** = 우리 수익 = 아고다 제휴 커미션. 예약금액의 약 4%($854K 중 $37K). **이것만 owner.**

**왜 전원 공개인가 (3가지):**
1. 예약금액은 우리 돈이 아니라 노출돼도 "회사가 얼마 버나"가 안 드러난다. 진짜 우리 수익(수수료 4%)만 가리면 영업비밀은 지켜진다.
2. 예약금액 없으면 성과 판단이 반쪽 — 건수만 보면 5성급 1건과 게스트하우스 5건이 같아 보인다. 금액이 있어야 "어디가 진짜 큰 예약을 만드나"를 안다.
3. 에디터가 봐야 일을 제대로 한다 — 예약·금액 큰 호텔로 다음 콘텐츠를 밀 수 있다. "내 영상이 이만큼 예약을 만들었다"는 동기부여도 된다.

**기존 원칙 갱신 근거:** 초기 성과표 API는 "스튜디오는 수수료·거래액을 **한 덩어리로** 비노출"로 짜서 예약금액까지 숨겼다. D-063에서 이미 둘을 갈랐고(예약금액=공개 / 수수료=owner), 오늘 그 방향을 구현에 반영. **수수료·거래액 뭉뚱그림 원칙 → 폐기.**

## 2. 화면 구성 (위 → 아래)

### ① 목적 배너 (맨 위)
"일·주·월 성과를 채널별로 5초에 → 잘되는 건 밀고 새는 건 고치러." 이 판이 왜 있는지를 화면 스스로 말한다.

### ② 기간 선택 — 글로벌 표준 + 비교
- **프리셋:** 오늘 · 어제 · 최근 7일 · 최근 30일 · 이번 달 · 최근 3개월 · 전체 · 기간 지정. (구글 GA4·쇼피파이 공통 표준)
- **비교 토글:** "이전 기간과 비교" — KPI에 `▲12%`·`▼3%p`로 지난번보다 나아졌나가 바로 보인다. "잘되나 못되나" 판단의 핵심.
- **원칙: 기간이 화면 전체(KPI·순위표·추세)를 지배한다.** 하나 누르면 통째로 그 기간으로 맞춰짐 → 5초 조망 성립.

### ③ KPI 요약 스트립
예약 · 확정률 · 취소율 · 예약금액(증감 표시) / 영상 수 · 조회수 · 클릭 · **[수수료 owner]**.
- 조회수·클릭 = 유튜브 연동 후 채워짐 → 지금은 정직하게 `—`(회색).

### ④ 렌즈 3개 (같은 데이터, 관점 전환)
채널별(기본) · 영상별 · 호텔별. **나라·언어는 렌즈 아님 = 필터.**
- **지금 실데이터:** 채널별(예약 7,169건 실집계).
- **연동 후 자리만:** 영상별·호텔별 — 예약↔영상(추적링크)·예약↔호텔(hid) 귀속이 아직 0건. 자리는 세우고 `연동 후`로 표시, 연동되면 같은 자리에 자동으로 실값.

### ⑤ 메인 순위표 (판단 → 행동)
- **기본 정렬 = 예약 건수순 + 순위 번호(#1·#2·#3).** 열 클릭으로 정렬 바꿈.
- 열: 채널 · 영상 · 조회수(`—`) · 예약 · 취소(건수·율) · 확정률 · 예약금액 · [수수료 owner].
- **행동 힌트:** 1등급 채널엔 "잘됨·더 밀기" 배지 / 취소 새는 채널엔 빨간 "호텔 ⑤로 고치러 →". 취소율 정렬 → 새는 곳 위로 → 눌러서 호텔 상세.
- 취소는 정렬용 신호만. 깊은 분석(왜·리드타임·TOP3)은 호텔 ⑤ 담당.

### ⑥ 드릴다운 4단계 (펼쳐보기)
채널 → 영상 → 소개된 호텔(TOP1·2·3) → 개별 예약. 리프 = breadcrumb + 호텔 요약바 + 개별 예약 표(매니저 D-061·호텔 ④ 컴포넌트 재사용). 영상·호텔 귀속은 연동 후 정확.

### ⑦ 월별 추세 (순위표와 나란히)
- **순위표와 다른 걸 답한다:** 순위표=어느 **채널**이 강했나 / 추세=어느 **달**이 강했나. 섞지 않는다.
- **기간에 연동:** 전체→월별 막대, 최근 30일→일별, 최근 3개월→주별(기간이 세분화를 정함). 막대 클릭 → 그 달/날 예약으로.
- 토글: 예약 · 예약금액 · 조회수(연동 후) · [수수료 owner]. 비교 켜면 이전 기간이 옅은 막대로 겹침.

### ⑧ 출처별 성과 비교 (D-066 연동 · 발행 후)
전략 추천 / 키워드 발굴 / 원고 유입 꼬리표를 발행 후까지 유지 → 출처별로 편수·예약·확정률 비교("데이터로 고른 게 진짜 더 잘됐나"). 발행이 쌓이면 채워짐.

## 3. 권한
- **예약금액·순위·추세·KPI(수수료 제외) = 전원(에디터 포함).**
- **수수료 KPI·열·추세 토글 = 대표님(owner)만** — 렌더 자체가 에디터 화면엔 안 뜸(D-053·D-061 원칙).

## 4. 데이터·연결
- 예약·취소·확정률·예약금액·수수료 = **bookings_agoda 직접 기간 집계**(기간 필터 위해 전체 집계 뷰 v_channel_stats 대신 원천을 기간으로 그룹핑). 확정률=실투숙 기준(is_completed, D-062 ⑤ A).
- 영상 수 = publications / 조회수 = D-059 후 / 클릭 = 추적링크 후.
- 수수료 = commission_usd, is_admin일 때만 응답 포함(서버에서 게이트).

## 5. 후속 / 재정리 조건
- 영상별·호텔별 렌즈 실데이터 = 추적링크·hid 귀속 연동 후.
- 출처별 비교 = 발행 시작 후.
- 조회수·클릭 = 유튜브 연동 후.
- 대표님 아고다 예약파일 업로드 시 예약금액·취소·확정률 재정리(D-062 ⑤와 동일 조건).

## 6. 데이터 갱신 주기 — 콘텐츠 축 vs 예약 축 (중요 · 2026-07-13 추가)

성과표엔 성격이 정반대인 두 데이터가 섞여 있다. **하나의 기간 필터를 둘에 똑같이 적용하면 안 맞는다.**

| 축 | 지표 | 갱신 | 기간(오늘·어제·최근 30일) |
|---|---|---|---|
| **콘텐츠 성과** | 조회수·클릭 | 실시간(백그라운드 배치가 하루 여러 번 자동 수집) | 살아있음(조회수 연동 후) |
| **예약 성과** | 예약·확정률·예약금액·수수료 | 대표님이 아고다 파일 업로드할 때만 | "○월 ○일까지" 누적 → 짧은 기간 대부분 0 |

### 예약 데이터 기준일 배지 (구현 완료)
- KPI 위에 `🕐 예약 데이터: {마지막 예약일}까지 반영 · 마지막 아고다 업로드 {업로드일}`.
- 데이터: API `data_asof` = max(booked_at)·max(created_at). (현재 = 2026-04-13까지 · 2026-07-10 업로드)
- 짧은 기간에서 0건이면 "예약 데이터는 ○일까지라 최근 기간은 비어 있어요. [전체로 보기]" 안내.
- **왜:** 대표님이 "오늘"을 눌러 0만 보고 "고장났나?" 오해하는 걸 막는다. 예약은 실시간이 아니라 업로드 스냅샷임을 화면이 정직하게 말한다.

### 조회수 = 세 숫자로 (D-059 나이별 수집과 연동)
- **① 누적 조회수** — 지금까지 총량. 항상 정확(모든 영상 최신 스냅샷 합).
- **② 기간 증가분** — 그 기간 늘어난 조회수("요즘 채널이 얼마나 보이나"). D-059 시계열 스냅샷 차이로 계산.
- **③ 그때 올린 영상 성적** — published_at 기준, 신작이 터졌나.
- **해상도 = D-059 수집 주기에 묶임:** 갓 올린 영상은 첫날 3회(발행 +3h/+12h/+24h)라 **오늘·어제 정밀** / 1개월 넘으면 월 1회라 "어제 하루 증가"는 못 내고 "이번 달 증가"까지. **∴ 오늘·어제 = 신작 성과에 강함, 최근 30일·이번 달 = 채널 전체 흐름에 유효.**

### 용량·비용 안전 (대표님 확인)
- 조회수 기록은 숫자 몇 개짜리라 가볍다. 나이별 수집이라 구영상은 월 1회만 늘어 **무한 증가하지 않음**.
- 규모 계산: 채널 30 · 영상 5만 · 10년 = 약 1천만 기록 ≈ 수백 MB~1GB (Supabase Pro 8GB 안에서 여유).
- 화면은 요약만 읽어(D-059 화면·수집 분리) 기록 수와 무관하게 빠름. 유튜브 호출은 화면과 무관(백그라운드 배치 · 무료 한도의 2% 미만). **화면 열 때 유튜브·아고다를 다시 부르지 않고 우리 DB만 읽는다 → 자주 봐도 비용 0.**
