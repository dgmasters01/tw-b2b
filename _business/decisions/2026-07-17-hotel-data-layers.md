## 2. 확정 — **뿌리는 하나. 밖의 것은 전부 다리**

```
                    hotel_code  (H-00001)          ← 🌳 뿌리. 절대 안 바뀐다
                         │
      ┌──────────────────┼──────────────────┬─────────────────┐
      │                  │                  │                 │
 agoda_hotel_ids    agoda_live_id      former_names      google_place_id
 [14개 · 다:1]      [파일에 실재]       [옛이름]          [구글]
 예약 리포트         지금 파는 번호       리브랜딩 추적      좌표·폐업
      │                  │
   예약 붙이기        링크·재고·분모
```

| 칸 | 뜻 | 누가 채우나 | 🔴 규칙 |
|---|---|---|---|
| **`hotel_code`** | **우리 고유번호 = 뿌리** | 우리 | **이름이 바뀌어도, 아고다ID가 갈려도 같은 곳.** 콘텐츠·성과표·키워드·매니저가 전부 이 위에서 돈다 |
| `agoda_hotel_ids` | 아고다ID **목록 (다:1)** | 예약 리포트 통합 | 실측 **454곳이 2개 이상 · 최대 14개**. **하나만 박지 않는다** |
| **`agoda_live_id`** 🆕 | **지금 살아있는 아고다 번호** | **파일** | 목록 14개 중 **파일에 실재하는 1개**. **뿌리 아님 — 다리** |
| `former_names` | 옛이름 이력 | 통합·파일 | 리브랜딩해도 같은 곳 |
| `operating_status` | 영업/폐업 | 구글 · 파일 | 폐업 호텔로 영상 만들면 예약 **0** |
| ~~`agoda_hotel_id`~~ | 🔴 **규격에 없다. 쓰지 않는다** | — | 2026-07-13 스키마 확정문에 없음. 오늘 클로드가 박았다가 되돌림 |

### 🔴 룰 3개 (이게 꼬임을 막는다)
1. **뿌리는 하나 — `hotel_code`.** 밖에서 온 번호(아고다·구글)는 **전부 다리**다. 뿌리로 삼지 않는다
2. **밖에서 온 것은 「목록」으로 받는다.** 아고다ID는 1개가 아니다. 하나만 박으면 13개를 잃는다
3. **밖의 데이터를 우리 칸에 덮어쓰지 않는다.** `agoda_inventory` 는 **따로** 둔다. `hotels` 는 **우리 명단**이다

## 3. 3층 — **각 층이 하는 일이 다르다**

| 층 | 무엇 | 주기 | 값 | 쓰는 칸 | 실패하면 |
|---|---|---|---|---|---|
| **① 파일** `agoda_inventory` | 전 세계 **304만** · 좌표·이름·주소·설명·사진·평점 | **월 1회** | **무료 · 호출 0** | `agoda_inventory` · `agoda_live_id` · 좌표·주소 | 옛 사진을 계속 쓴다. **화면이 날짜를 말한다** |
| **② 아고다 API** | *"이 호텔 **지금** 파나 · 얼마"* | **1건 · 필요할 때** | 무료 · 한도 미확인 | 안 씀(읽기) | 파일 값으로 보여주고 **"확인 못 함" 표시** |
| **③ 구글 Places** | *"폐업했나"* · 파일에 없는 곳 | **가입 · 영상 만들기 직전** | **유료 · 하루 150** | `operating_status`·`google_place_id` | 건너뛴다. **막지 않는다** |

**월 1회가 맞는 이유** (대표님): 호텔 재고는 **매일 안 바뀐다.** 신규·폐업은 월 단위다.
**"전체를 실시간"은 불가능하고 필요도 없다.** 실시간이 필요한 건 **「이 호텔 하나」** 이고 그건 ②③이 한다.

🔴 **파일은 사진 한 장이다.** `agoda_live_checked_at` 에 **언제 찍은 사진인지** 남긴다. 화면이 그 날짜를 말한다(㊶-6 잣대 도장).

## 6. 검사 — **매번 이걸 통과해야 한다**

```sql
-- ① 매니저 호텔이 안 다쳤나 (봇 돌린 뒤 반드시)
select count(*) from hotels where user_id is not null and status not in ('pending','approved','paid');
-- 0 이어야 한다

-- ② 뿌리가 안 바뀌었나
select count(*) from hotels where status='catalog' and hotel_code is null;
-- 0 이어야 한다

-- ③ 한 아고다ID 가 두 마스터에 붙지 않았나 (예약이 두 번 세어진다)
select x.aid, count(distinct hotel_code) from hotels h,
  lateral jsonb_array_elements_text(h.agoda_hotel_ids) x(aid)
 group by 1 having count(distinct hotel_code) > 1;
-- 0행이어야 한다
```

**룰: 돈에 닿는 표를 만지는 봇은 「무엇을 안 만지는지」를 코드에 적는다.** 무엇을 만지는지만 적으면, 다음 클로드가 한 칸 더 만진다.

### 실측 — 전부 **「같은 호텔인데 이름이 조금 달라서 마스터가 쪼개진 것」**

**이게 정확히 대표님이 고유번호를 만드신 이유다** — *"호텔이 이름이 바뀌거나 할 때 같은 곳으로 보기 위해서"*.
자동 통합이 **이름으로 묶다가 못 묶은 것**이고, **아고다ID 가 「같은 곳」이라고 알려주고 있다.**

### 🔴 무엇이 잘못되나
같은 아고다ID 가 마스터 3곳에 붙어 있으면 그 호텔의 예약이 **성과표·키워드·지역 순위에서 3번 세어진다.**
지금 화면의 **난바 588건 · 오사카 1,081건에 이 오염이 섞여 있다.**

### 확정 — **파일이 이걸 푼다**
파일은 아고다ID 별로 **한 줄**이다. `agoda_live_id` 를 채우면 **같은 `agoda_live_id` 를 가진 마스터 = 같은 호텔**이 드러난다.
| | |
|---|---|
| **자동으로 합치나** | 🔴 **안 한다.** 3,182곳 중 84곳이다. 합치면 `hotel_code` 가 사라지고 **콘텐츠·성과표가 딴 호텔을 가리킨다**(룰 M3) |
| **어떻게** | `merge_status='ambiguous'` 로 표시 → **`api/hotel-review.js` 가 이미 그걸 어드민에게 보여준다**(2026-07-13 결정 3-3 *"검증 로봇이 애매한 묶음만 표시 → 점진 정리"*) |
| **누가 정하나** | **대표님.** 예약 많은 것부터 (2026-07-13 결정 3-2 *"상위 100~200부터 대표님이 눈으로 확정"*) |

**설계가 이미 이걸 예상하고 있었다.** 검증 로봇 자리(`hotel-review.js`)도 있고, 상태 칸(`merge_status`)도 있다. **비어 있던 건 「검사」뿐이었다.**

**룰: 검사를 먼저 만든다. 검사가 없으면 「깨진 줄도 모른다」.** 45건이 3주째 조용히 예약을 부풀리고 있었다.
