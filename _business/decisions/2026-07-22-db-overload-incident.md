# 🔴 2026-07-22 사고 — 아고다 자료 과적으로 서비스 중단 (D-073)

## 무슨 일

`agoda_inventory` 를 «예약 있는 도시 전부(166곳)»로 담고, 언어별 이름까지 4개 언어를 넣었다.

| | 아침 | 사고 시점 |
|---|---|---|
| DB 크기 | 약 200MB | **835MB** |
| agoda_inventory | 61,514행 | 362,436행 (418MB) |
| agoda_inventory_name | 5,419행 | 약 145만행 (361MB) |

Supabase **무료 요금제 · NANO(메모리 0.5GB · DB 한도 500MB)** 를 넘겼다.
데이터가 메모리에 안 들어가니 모든 질의가 디스크로 가고, **Disk IO Budget 이 소진**되어
디스크가 5MB/s 로 묶였다. 결과: **로그인 36초 → 504. 서비스 몇 시간 중단.**

## 원인 (전부 클로드 잘못)

1. 참고 자료(아고다 300만 곳)를 **사업 DB 에 통째로** 넣었다.
2. 넣기 전에 **그릇 크기(무료 500MB)를 확인하지 않았다.**
3. 보호 장치였던 `db-query` 시간당 60회 한도를 **600으로 풀고** 그 위에 대량 적재를 몰아쳤다.
4. 데이터가 다 들어간 뒤에도 **"작동 확인"을 이유로 워크플로를 한 번 더 돌렸다.**

## 복구

프로젝트 재시작(연결이 꽉 차 SQL 조차 안 들어갔다) → `truncate agoda_inventory_name` · `truncate agoda_inventory`
→ **835MB → 56MB**, 로그인 36초 → **0.4~1.5초**. 사업 데이터(호텔 3,185·예약 7,316·발행 3·회원 5·도시 25,268) **손실 0**.

## 🔴 규칙 (앞으로 만들 모든 서비스에 적용)

> **참고 자료는 사업 DB 에 살지 않는다.**

| 종류 | 어디에 | 예 |
|---|---|---|
| **사업 데이터** | DB · 작게 · 백업 필수 | 호텔·예약·회원·계약·발행 기록 (전부 합쳐 **40MB**) |
| **참고 자료** | DB 밖 · 필요한 것만 추려서 | 아고다 300만 곳, 크롤링 결과 |

- 대량 적재 전 **반드시 `pg_database_size` 와 요금제 한도를 먼저 잰다.**
- 참고 자료를 담을 땐 **«우리 것»만**: 호텔 마스터 아고다ID ∪ **예약에 나온 아고다ID(취소 포함)**.
  (대표님 지적: *"예약취소 호텔까지 하면 호텔 수는 더 많아져야 되는 거 아니야?"* — 맞다. 취소돼도 검색·클릭된 호텔이다.)
- **무거운 칸은 안 담는다**: `overview`(소개글)·`photo1`(사진주소) — 판정에도 키워드에도 안 쓴다.
- `db-query` 한도는 **보호 장치다. 작업이 막힌다고 푸는 것이 아니라 작업을 쪼갠다.** (600 → 120 원복)

## 도구 반영

`_os/tools/agoda-file-load.py --step inventory` 를 **«우리 호텔만»** 방식으로 교체(커밋 `7715fe5f`).
→ 월 1회 자동 갱신(`.github/workflows/agoda-inventory-refresh.yml`)이 돌아도 **약 4,000곳 · 5MB** 만 담는다. 다시 터지지 않는다.
