## ② 이유

**라운드 13~14 합의 그대로 박은 이유:**
- 기본값 세금계산서: B2B 타겟 + 한국 세무 실무 기본 (라운드 14 직접 확정)
- 3종 분리: 한국 세법상 사업자/개인 영수증 트랙이 다르므로 시스템 단에서 분기
- B2B 안내 문구 옆 작게: 고객이 자연스럽게 ①선택하도록 유도 + 개인 결제는 예외임을 가이드

**프론트만 박은 이유 (백엔드 손 안 댄 이유):**
- 라이브 점검 결과 `api/invoice.js` handleIssue가 이미 `kr_receipt_type` + `kr_receipt_meta` + `bill_to_business_no` 받을 준비 완료 (line 273~346)
- 3종 타입 검증·기본값 `tax_invoice`·메타 박기 모두 박혀있음
- invoices 테이블에 `kr_receipt_type` / `kr_receipt_meta` / `kr_receipt_issued` / `kr_receipt_issued_at` 컬럼 박혀있음
- **백엔드는 단계 4~5에서 이미 박혔고 프론트만 빠짐** → 프론트만 박으면 100%

**동적 폼 분리 함수 (`renderReceiptFields`):**
- 라디오 변경 즉시 폼 자동 갱신
- 폼 코드 한 곳 모음 → 향후 새 영수증 종류 추가 시 한 곳만 손댐

**필수 필드 검증 클라이언트 + 서버 이중:**
- 클라이언트: 누락 시 invalid 빨강 + 빠른 피드백 (UX)
- 서버: handleIssue가 이미 `invalid_kr_receipt_type` 응답 박음 + Supabase 컬럼 제약
- 사고 방지 이중 안전망

**휴대폰번호 숫자만 추출:**
- `010-1234-5678` / `01012345678` 둘 다 입력 가능
- 서버로 보낼 때 숫자만 (정규화)
- 10~11자리 검증으로 잘못된 입력 차단

## ③ 사업 영향

**한국 세무 의무 자동 충족 시스템 가동.**

이 코드가 박힌 후 한국 매니저가 결제할 때:

**시나리오 A — 일반 호텔 법인 (예: 신라호텔):**
1. sales.html 진입 → 🇰🇷 KRW 송금 카드 + 영수증 섹션
2. 세금계산서 (DEFAULT) 자동 선택됨
3. 사업자번호·상호·대표자명·이메일 입력 (필수 4개)
4. "📄 인보이스 발행하기" 클릭 → INV-KR-2026-0001 발행 + kr_receipt_type=tax_invoice + kr_receipt_meta 박힘
5. 입금 확인 후 대표님이 홈택스에서 전자세금계산서 발행 → admin "발행 완료" 체크 (BL-INVOICE-002 영역)

**시나리오 B — 개인사업자 호텔:**
1. 라디오에서 "현금영수증 (사업자 지출증빙)" 선택
2. 사업자번호만 입력 (간소화)
3. 발행 → kr_receipt_type=cash_receipt_business

**시나리오 C — 개인이 송금 (드문 케이스):**
1. 라디오에서 "현금영수증 (개인 소득공제)" 선택
2. 휴대폰번호 입력
3. 발행 → kr_receipt_type=cash_receipt_personal

**한국 세무 의무 100% 충족:**
- 사업자에게는 전자세금계산서 (한국 세법 표준)
- 사업자가 현금영수증 원하면 지출증빙
- 개인이 소득공제 원하면 개인 현금영수증
- 영수증 종류가 인보이스 PDF에 미리 박혀 고객이 발행 직전 확인 가능

**미구현 남은 영역 (AGR-0013 partial 사유):**
- admin-invoices에서 영수증 "발행 완료" 체크 + 24시간 미발행 시 텔레그램 알림 → BL-INVOICE-002 신설 시 박음 (AGR-0005와 통합)
