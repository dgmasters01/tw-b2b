## ② 이유

- **단계 5 정합성 정정 동시 처리**: 단계 9 chat-log에서 "단계 10에서 일괄 처리" 박음. 매니저 본인이 dashboard에서 자기 PDF 다운받으려면 pdf 액션이 본인 인증으로도 통과해야 함. router에서 admin → authed fallback이 정석 — handlePdf/handleGet 함수 본체는 손대지 않음(이미 `isOwnerOfInvoice` 검사 박혀있음).
- **my-list 별도 액션 박은 이유**: 단계 6의 `list`는 admin 전용. 매니저 본인은 admin이 아님 → 별도 액션 필요. my-pending과 분리해서 박은 이유: 두 흐름의 사용처가 명확히 다름 (my-pending = sales 차단 모달용, my-list = dashboard 서류 탭). 합치면 분기 if문 늘어남.
- **invoice-engine 추출 미루는 이유 (단계 7 chat-log 결정 정정)**: paypal.js의 자동 발행 흐름 vs invoice.js의 admin 발행 흐름이 형태가 너무 다르고(메타 박는 방식, 시점, 에러 처리 모두 다름), 추출하면 양쪽 다 손대야 함 = 단계 10 범위 폭발. **별도 BL(BL-INVOICE-ENGINE-EXTRACT)로 분리해서 후속 정리**. 단계 10은 manager-dashboard가 본질이라 거기 집중.
- **카드형 UI 선택**: 단계 6의 admin-invoices는 테이블(컬럼 10개) 형태였지만, 매니저 dashboard는 모바일 우선이라 카드형. 각 행에 핵심 정보(인보이스 번호/호텔/금액/상태/PDF 버튼)만 박힘.
- **빈 상태 안내 차등**: status='all'에서 0건 = "아직 발행된 문서가 없습니다" / 특정 status에서 0건 = "해당 상태의 문서가 없습니다". 매니저가 헷갈리지 않게.

## ③ 사업 영향

- **매니저가 본인 회계·증빙 자료를 즉시 확보**: 결제하면 manager-dashboard '서류' 탭에서 인보이스/영수증 PDF가 자동 누적. 매니저가 따로 이메일 검색·문의 안 해도 됨 = 대표님 고객 지원 부담 감소.
- **연간 인보이스 전체 추적 가능**: 매니저가 6개월 후 환불 보장 행사할 때 자기 결제 내역을 한 화면에서 확인. "영수증 어디 갔지" 사고 0.
- **세무 신고 자료 자동 정리**: 매니저 본인이 KR 한국 호텔이면 자기 KRW 인보이스 + 세금계산서 발행 예정 정보가 함께 보임. 한국 매니저는 세무사에게 PDF 1장만 넘기면 끝.
- **i18n 자동**: 한국 매니저는 한글, 글로벌 매니저는 영문. 같은 화면이 매니저 국적별로 자동 분기.
- **모바일 우선**: 카드형 UI + 반응형 — 매니저가 핸드폰으로 manager-dashboard 들어가도 PDF 1클릭 가능.
