## ② 이유

- **정책 2.14 "5년 영구 보관"의 시스템화**: 한국 상법·법인세법은 사업 관련 문서 5년 보관 의무. 매니저 환불 보장 6개월 후에도 인보이스 추적 가능해야 함. retention_until 컬럼 박지 않으면 향후 "이 인보이스 언제까지 보관해야 하지?" 매번 사람이 계산해야 함.
- **trigger 자동 박음 이유**: 단계 4 handleIssue / 단계 7 PayPal 자동 발행 / 향후 다른 발행 경로 모두 retention_until 박지 않아도 자동으로 +5년 박힘. 코드 다 손댈 필요 없음. 멱등 보장(NULL일 때만 박음).
- **view로 통합한 이유**: invoices + credit_notes를 매번 UNION 쿼리 짜야 하는 부담 제거. 향후 admin 화면에 retention_state 표시할 때 view 1번 SELECT로 끝남.
- **자동 삭제 박지 않는 결정 (의도적)**:
  - 정책 2.14 "5년 영구 보관" 문구 그대로 — 자동 삭제는 정책 위반 가능성
  - 한국 세무 분쟁 시 5년 지났더라도 증빙 필요한 케이스 있음 (소송 시효 등)
  - expired 발견 시 텔레그램 알림으로 대표님 결정 대기 → 별도 BL(BL-INVOICE-ARCHIVE-DELETE)로 갈무리
  - 자동 삭제 박았다가 잘못 삭제하면 되돌릴 수 없음 (헌법 9 가역성 위반)
- **GitHub Actions cron 선택**: vercel.json crons는 Hobby 한계(매일 1회만), Actions가 정석. 단계 8과 동일 패턴.
- **매일 1회 (매시 아님)**: retention 상태는 빠르게 변하지 않음 — 매일 1번이면 충분. cron 부하 최소화.
