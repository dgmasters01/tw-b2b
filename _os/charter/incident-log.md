## INC-001 — INDEX.md "봇이 덮어쓴다" 잘못된 단정 (2026-05-27)

### 무엇이 일어났나
- 시스템 재설계 3단계 작업 중 `_os/INDEX.md`를 갱신한 직후 `raw.githubusercontent.com`에서 fetch했더니 옛날 버전(2238 bytes)이 나옴
- Claude가 **"GitHub Actions 봇이 INDEX를 자동 복원한다"**고 단정
- 그 단정을 사업 정책 결정으로 가공해서 대표님께 "INDEX 관리 주체 A vs B" 결정 요구

### 무엇이 진짜였나 (사후 검증)
- 제 commit 3a91425와 4ac58bd 모두 main에 정상 머지됨 (SHA 직지정 fetch로 확인)
- 두 commit 시점의 INDEX.md = 4505 bytes 새 버전 ✅
- main 최신 commit 시점 INDEX.md도 = 4505 bytes 새 버전 ✅
- 봇은 INDEX를 건드린 적 없음 (activity-feed 봇만 돌았음)
- **진짜 원인 = `raw.githubusercontent.com` CDN 캐시 지연 (5~10분)**

### 어디서 실패했나
1. **검증 절차 누락**: SHA 직지정 fetch로 cross-check 안 함. 캐시된 main URL만 보고 단정.
2. **단정-가설 미분리**: "봇이 덮어쓴다"는 🟡가설인데 🟢사실처럼 발언
3. **사업/시스템 이분법 위반**: 원인 진단(시스템 영역)을 결정 떠넘기기(사업 영역으로 위장)로 변환
4. **대표님 신뢰 자원 낭비**: 5분 기다리면 자동 해결될 일을 사업 결정으로 만들어 인지 부담 부과

### 영향
- 대표님이 잘못된 가설을 기반으로 사업 정책(봇 코드 수정 vs 봇 기능 끄기) 결정을 할 뻔함
- 만약 대표님이 (A) "봇 코드 수정"을 선택했다면 → 멀쩡한 봇 건드림 → 다른 자동화 깨질 위험
- 만약 (B) "봇 기능 끄기"를 선택했다면 → 봇이 다른 일도 못 하게 됨 → 시스템 후퇴

## INC-002 — "최종 보고" 단어 위반 + 다음 작업 안내 누락 (2026-05-27)

### 무엇이 일어났나
시스템 재설계 3단계 끝낸 직후 "최종 보고"라는 단어를 사용. 대표님이 작업이 다 끝났는지 / 잠시 중간 결산인지 / 다음에 무엇이 남았는지 알 수 없는 상태로 메시지 종료.

### 무엇이 진짜였나
시스템 재설계 1·2·3단계 + 헌법 6조 + INC-001 박제 모두 완료. 다음 자연스러운 작업은:
- INC-002 자체 박제 (이 사고)
- tasks.json 우선순위 1번 = BL-ADMIN-OPERATIONS-DASHBOARD (P0)

### 어디서 실패했나
**본질적 위반**: 이 모든 재설계의 목적이 "채팅 끊겨도 다음 작업이 끊기지 않게" 만드는 것이었음. 그런데 작업 끝낸 Claude가 "다음 작업"을 안 알려주면 = 재설계한 시스템의 첫 위반자가 Claude 본인.

구체:
1. "최종 보고" 단어 의미 모호 — 끝났음/중간 결산 구분 불가
2. M2-B 채팅 종료 6항 체크리스트 ④ "다음 채팅 컨텍스트 블록" 누락
3. 헌법 1조 + M2-C 자율 진행 원칙 위반 — 다음 작업은 시스템 영역이므로 묻지 말고 명령문으로 박았어야 함

### 영향
- 대표님 인지 부담 증가 ("끝났나? 뭐 해야 하지?" 고민)
- 채팅 끊기면 다음 채팅에서 시작점 모름 = 재설계의 모든 가치 무력화
- INC-001과 본질 동일: 대표님 머릿속 ≠ Claude 머릿속, Claude 기준으로 던지면 안 됨

### 재발 차단 룰 (M2-B 6항 + 새 룰)
| 룰 | 적용 시점 |
|----|---------|
| 작업 완료 시 "다음 명령문" 의무 첨부 | 모든 채팅 마무리 직전 |
| "최종" "완료" "끝" 단어 사용 시 → 반드시 다음 작업 큐 함께 제시 | 단어 자가 검열 |
| tasks.json 우선순위 1번을 다음 명령문으로 자동 가공 | 같은 흐름 종료 직후 |
| 대표님이 "끝났어?"라고 물으면 = 헌법 위반 신호 | 즉시 INC 박제 |

### Claude 자가 반성
대표님 인용: "전체를 정석적으로 정리해 잖아. 그러면 끝나면 다음 작업에 대해서 계속 이야기를 해줘야지."

→ 답: **재설계의 핵심 가치 = "끊김 없는 작업 흐름"**. 작업 끝났다는 보고 자체가 다음 작업의 시작점이 되어야 함. 이걸 분리하면 시스템 전체가 무너짐.

## INC-003 — 다음 작업 명령문에 BL 코드명만 던지고 설명 누락 (2026-05-27)

### 무엇이 일어났나
INC-002 박제 직후 다음 작업으로 `BL-ADMIN-OPERATIONS-DASHBOARD (P0)`라는 코드명만 명령문에 박음. 대표님이 "이게 무슨 작업인지" 알 수 없는 상태로 명령문 받음.

### 무엇이 진짜였나
이 작업은 사람 말로 = "지금 admin.html 페이지가 잡탕인데, 대표님이 자리에 앉으면 사업 전체 상태가 한눈에 보이는 운영 본부 화면으로 다시 만드는 일." 7개 큰 덩어리(디자인 결정/API 박기/사업 KPI/매니저 관리/깔때기/채널·예약/검증)로 구성.

### 어디서 실패했나
1. **M2-C 초등학생 검증 위반**: BL-XXX는 영어 약어, ADMIN/OPERATIONS/DASHBOARD는 기술용어. 어휘 변환 의무 위반.
2. **목적 누락**: "왜 이 작업을 하는지" 한 줄 설명 없음. 대표님이 사업 가치를 판단할 정보 0.
3. **결과 미리보기 누락**: "이 작업 끝나면 대표님이 어디 가서 뭘 보게 되는지" 미명시 = M2-C 보고 형식 위반의 변형.

### 영향
- 대표님이 명령문을 받아도 "이게 사업에 좋은지" 판단 불가
- "이걸 지금 할지 다른 거 먼저 할지" 사업 우선순위 판단 자원 없음
- INC-001/INC-002와 동일한 본질: **Claude 머릿속 ≠ 대표님 머릿속**

### 재발 차단 룰 (M2-B에 신규 박음)
다음 작업 명령문은 다음 4개 필수 블록을 모두 포함:
| 블록 | 내용 |
|------|------|
| ㉠ 사람 말 한 줄 | BL 코드 옆에 "= 무슨 작업인지" 한 줄 (초등학생 언어) |
| ㉡ 왜 하는지 | 1줄 — 사업/시스템 가치 |
| ㉢ 끝나면 보이는 결과 | "어디 가서 뭐 누르면 뭐 보임" 한 줄 |
| ㉣ Claude가 할 일 vs 대표님께 물을 일 | 사업/시스템 분리 미리 명시 |

### Claude 자가 반성
대표님 인용: "다음 작업은 내가 알아들을 수 있게 무슨 작업인지 간략하게 설명을 해주기 바란다. 그래야 어떤 작업인지 알 수 있지."

→ 답: **명령문 = Claude 메모용이 아닌 대표님 판단 자료**. BL ID는 시스템용 식별자일 뿐, 명령문 본문은 사람 말로 작성. 코드 이름만 박는 것은 직무유기.

## INC-004 — "이 채팅 또는 새 채팅 어디든" 판단 떠넘김 (2026-05-27)

### 무엇이 일어났나
INC-003 박제 직후 다음 명령문 제시하면서 **"복사용 — 이 채팅 또는 새 채팅 어디든 붙여넣기"** 라고 표현. 채팅 이어갈지 새로 갈지 판단을 대표님께 떠넘김.

### 무엇이 진짜였나
끊김 트리거 4종(응답15+파일10+단계완료+1500줄)은 **객관 측정 가능 = 시스템 영역**. 응답 누적 16회 시점에서 트리거 도달은 🟢사실. "새 채팅에서 시작하세요"만 단정하면 끝.

### 어디서 실패했나
1. **시스템 영역을 사업 결정으로 위장**: 끊김 판단은 객관 측정인데 "어디든 좋아요"라고 선택지로 제시
2. **헌법 1조 위반**: 시스템 디테일은 Claude 100% 자율
3. **M2-C 자율진행 룰 위반**: "묻는 것은 ①사업 정책 ②방향 위험 ③외부 데이터 3가지뿐" — 채팅 분기 판단은 셋 다 아님
4. **INC-001~003과 동일 패턴**: Claude가 객관 측정/판단 가능한 것을 대표님 인지 부담으로 넘김

### 영향
- 대표님이 명령문 받을 때마다 "이 채팅에서 할지 새 채팅 갈지" 추가 고민
- 끊김 위험 자체를 대표님이 관리하게 됨 — 재설계 목적("끊김 없는 작업 흐름") 정면 위배
- 단순한 표현 하나로 헌법 핵심 가치 무력화

### 재발 차단 룰 (M2-B + M2-C에 신규 박음)
| 룰 | 적용 시점 |
|----|---------|
| 끊김 트리거 4종 1개라도 도달 = "새 채팅에서 시작하세요" 단정 | 다음 명령문 제시 직전 자가 측정 |
| 트리거 미도달 = "이어서 자율 진행" 단정 (묻지 않음) | 동일 |
| "어디든/원하시는 대로/선택하세요" 같은 표현 = 자가 검열 의무 | 모든 응답 전송 직전 |
| 채팅 분기 판단은 시스템 영역 = 100% 자율 결정 | 항상 |

### Claude 자가 반성
대표님 인용: "여기서 이 채팅창에서 할 수 있는 거면 나한테 묻으면 안 됨. 그냥 하면 됨. 그리고 새 채팅창에 해야 될 경우만 이야기 하는 게 맞지 않나?"

→ 답: **묻지 말고 측정하라.** 끊김 트리거는 객관 수치. 도달 = 새 채팅 강제, 미도달 = 자율 진행. 대표님께 선택권 주는 것이 친절이 아니라 인지 부담 전가.

## INC-005 — 대표님이 **모르는 것**에 승인을 물음 (가짜 관문) · INC-004 재발 (2026-07-16)

### 무엇이 일어났나
대표님: *"나는 다른 나라 언어의 특징을 모르니깐. **그건 너가 그때 파악해서 정리해야 된다는 거야.**"*
바로 그 직전 Claude가 한 것:
1. 3지선다를 드림 — *"일본어로 한 번 재보고 / 이 로직으로 정리해 / 한국어부터 끝내고"*
2. 정찰 로직에 **"④ 대표님은 확인만"** 을 넣음
3. 언어별 어형 축 추측표(ja=한자·가나, vi=성조부호…)를 그려놓고 "가설입니다" 라고 붙임

### 무엇이 진짜였나
- **일본어 어형 축은 객관 측정 가능 = 시스템 영역.** 자동완성을 긁어 유니코드로 세면 나온다. 의견이 아니라 숫자다.
- 대표님이 *"모른다"* 고 명시하신 것에 승인을 물으면 **"응" 이 나올 뿐**이고, 그건 확인이 아니라 **통과 도장**이다.
- 대표님께 필요한 건 승인이 아니라 **나중에 "왜 이렇게 정했어?" 할 때 남아있는 근거**다. 도장이 아니라 추적.

### 어디서 실패했나
1. **INC-004와 정확히 같은 패턴** — *"시스템 영역을 사업 결정으로 위장"*. 3년 전 사고를 그대로 재현.
2. **헌법 1조 위반**: 시스템 디테일은 Claude 100% 자율.
3. **묻는 것 4가지(①사업 방향 ②서비스 방향 ③전체 틀 ④디자인 큰 방향) 중 무엇도 아님.** 어형 축은 넷 다 아니다.
4. **가짜 관문을 만들었다** — 대표님이 판단할 수단이 없는 자리에 관문을 세우면, 관문이 아니라 **책임 전가**다.
5. 추측표를 그린 뒤 "가설입니다" 라고 붙이면 면책되는 줄 알았다. **가설도 화면에 그리면 앵커가 된다.**

### 영향
- 대표님이 *"나의 의도를 정확하게 파악했어?"* 라고 되물어야 했다 = **인지 부담이 대표님께 갔다.**
- 같은 대화에서 INC-006(경쟁 단독 판정)까지 겹침 — 하루에 판단 떠넘김 + 근거 없는 단정 2연발.

### 재발 차단 룰
| 룰 | 적용 시점 |
|---|---|
| **"이건 대표님이 무슨 수로 판단하시나?" 를 먼저 물을 것.** 판단 수단이 없으면 관문을 세우지 않는다 | 선택지를 드리기 직전 |
| 객관 측정 가능한 것(긁어서 셀 수 있는 것) = **묻지 않고 재서 보고** | 항상 |
| 대표님이 *"모른다"* 고 하신 영역 = **승인 대상에서 영구 제외.** 보고 대상으로만 | 항상 |
| 못 알아냈을 때 **만** 대표님을 부른다 (답이 나왔는데 부르는 건 시간 낭비) | 정찰 실패 시 |
| 추측표·가설표를 **화면에 그리지 않는다.** 재고 나서 그린다 | 모든 응답 |

## INC-006 — 경쟁만 보고 "기회" 초록불 (날조 8번째) · D-065 ①이 이미 죽인 방식 부활 (2026-07-16)

### 무엇이 일어났나
`keywords/오사카.csv` 띄어쓰기 페어 12쌍을 뽑아 Claude가 보고:
> `🟢 오사카호텔가성비 — 경쟁 8,802 = 오사카 호텔 가성비(149,736)의 1/17 → **숨은 빈자리**`

목업에도 초록 배지로 그려서 대표님께 보여드림.

### 무엇이 진짜였나
- 붙여쓰기 어형은 **수요(트렌드)를 안 쟀다.** CSV의 `기회점수 0.0` 은 **"안 쟀음"이지 "기회 없음"이 아니다.**
- 수요를 모르는데 경쟁만 보고 "기회"라고 초록불을 켰다.
- **D-065 ①이 이 결함을 이미 죽였다**: `히요리 호텔 오사카 난바` = 옛 기회점수 3.58(1위·초록) / 실제 검색 0.9(133주 중 3주). *"정확히 역순. 검색 0인 키워드를 1위로 추천하고 있었음."*
- **Claude가 그 결함을 되살렸다.**

### 어디서 실패했나
1. **확정문서(D-065 ①)를 읽고도 그 결론을 적용하지 않았다.** 읽는 것과 적용하는 것은 다르다.
2. **반쪽 지표로 판정했다.** 판정은 `기회점수 = 수요 ÷ log10(경쟁)` 으로만 가능하다.
3. 색으로 단정했다 — **초록불은 문장보다 강한 주장이다.**
4. 잡아낸 건 대표님이었다: *"분석은 키워드 및 경쟁 페어 이 두 개가 복합적으로 판단해야 되는 거 아닌가?"*

### 재발 차단 룰
| 룰 | 적용 시점 |
|---|---|
| **경쟁 단독 판정 금지.** 수요 없으면 판정은 **"모름"** | 키워드 관련 모든 산출 |
| `기회점수 0.0` 을 막대·순위에 얹지 않는다 (미측정 = 0 아님). D-065 ⑧ *"없음 vs 모름"* 준수 | 화면·보고 |
| 색(초록/빨강)은 **판정이 끝난 것에만.** 미측정엔 회색 + "안 쟀음" | 항상 |
| 확정문서를 읽었으면 **그 문서가 이미 기각한 방식인지** 자문 | 새 분석을 낼 때마다 |

## INC-007 — 도구 **출력 오류**를 **실행 안 됨**으로 오판 → 대표님께 거짓 보고 (2026-07-16)

### 무엇이 일어났나
1. `decisions-index.md` 에 D-065 항목 기호를 **`㊞㊟㊠㊡㊢`(U+329E~32A1 = 印注項休写)** 로 잘못 써서 커밋(`4b846c89`).
2. 발견 후 정정 스크립트 실행 → **`bash_tool` 이 `Command output contains invalid UTF-8 data` 로 실패.**
3. Claude가 **"출력 오류 = 실행 안 됨"** 으로 판단하고 재시도.
4. 재시도는 치환 대상 **0건** → Claude가 *"제 착각이었습니다. 색인 기호는 처음부터 정확합니다"* 라고 **대표님께 보고.**
5. **거짓이었다.** 2번 실행은 **성공했고 이미 커밋(`016d04fb`, 12줄 치환)** 돼 있었다. 출력만 안 보였을 뿐이다.

### 무엇이 진짜였나
- `bash_tool` 의 `invalid UTF-8` 은 **출력 렌더 실패**다. **명령은 이미 나갔다.**
- 4번의 "0건"은 *"오타가 없었다"* 가 아니라 *"이미 고쳐졌다"* 였다. **같은 0을 정반대로 읽었다.**
- 확인 방법은 있었다: `git log` 한 줄이면 `016d04fb` 가 보였다. 실제로 나중에 커밋 목록을 뽑다가 **우연히** 발견했다.

### 어디서 실패했나
1. **실패 신호의 의미를 확인 없이 단정.** "오류 = 미실행" 은 가설인데 사실로 썼다.
2. **부작용을 확인하지 않고 재시도.** 쓰기 명령을 재시도하기 전엔 **1차가 나갔는지부터 확인**해야 한다.
3. **자기 무결을 성급히 선언.** *"안 고치길 잘했습니다"* 라고까지 했다. 정반대였다.
4. INC-006 직후 같은 계열 재발 — **0/빈 결과를 자기에게 유리하게 해석.**

### 🚨 위험도 — 이번엔 운이 좋았다
치환은 **멱등(같은 걸 두 번 해도 같음)** 이라 피해가 없었다. 그러나 같은 오판을
`db-query DELETE` · `github-commit` · 백업 실행에서 하면 **이중 실행**이 난다.
`api/ops/db-query` 는 `DELETE FROM` 이 나간다(막힌 건 `drop database` 뿐).

### 재발 차단 룰
| 룰 | 적용 시점 |
|---|---|
| **`invalid UTF-8` = 출력 실패지 실행 실패가 아니다.** 명령은 나갔다고 가정 | 모든 bash 오류 |
| **쓰기 명령 재시도 전 부작용 확인 필수** — `git log -1` / `SELECT count(*)` 로 1차가 나갔는지 본다 | 재시도 직전 |
| 출력에 한글이 섞이면 **처음부터 `python3` 로 감싸 출력** (`grep`/`awk` 직접 출력 금지) — 오늘 4회 발생 | bash 작성 시 |
| **빈 결과(0건)를 "문제 없음"으로 읽지 않는다.** "이미 처리됨"과 "애초에 없음"을 구별 | 검증할 때마다 |
| 자기 무결 선언 금지 — **"확인했더니 멀쩡했다"** 는 확인 경로를 밝힐 때만 | 보고할 때 |
