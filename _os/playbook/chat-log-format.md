# Playbook — chat-log 작성 표준 (사업가 시점 5블록)

> **헌법 6조 (AI 가독성) 이중 형식 의무를 chat-log 단위까지 확장.**
> 사람용 = 사업가 5블록 / 기술 상세 = 그 아래로 분리.

---

## 1. 의무 5블록 — 사업가 시점

모든 chat-log는 frontmatter 닫힘(`---`) 직후 다음 5블록을 **반드시 순서대로** 포함한다:

```markdown
---
slug: ...
title: ...
date: ...
tasks: [...]
commits: [...]
decisions: [...]
---

## 🎯 한 줄 요약
(비기술 용어, 1줄. "무슨 일이 있었나"를 사업가 언어로)

## 📍 왜 발생했나
(2줄 이내. 사업적 배경 또는 사용자 영향. "기술 부채" 같은 개발자 용어 금지)

## 🛠 어떻게 해결했나
(2~3줄. 사업가가 이해 가능한 수준으로. "API 함수 통합 / 화면 갱신 / 결제 연동" 등)

## ✅ 결과
(완료된 사업 가치. "매니저가 한 번에 본다 / 결제 화면이 빨라졌다" 같은 결과 중심)

## ⏱ 다음 결정 필요
(있으면 1~2줄, 없으면 "없음"이라고 명시)

---

# 🔧 기술 상세 (개발자용)

(여기서부터는 commit hash, task ID, Phase, 파일 경로, 코드 블록 등 자유롭게)
```

## 2. 용어 변환 표 (사업가 시점에서 사용 금지 → 대체어)

| ❌ 개발자 용어 | ✅ 사업가 언어 |
|---|---|
| commit / 커밋 | 작업 / 작업 단위 |
| task / 태스크 / Task ID | 업무 / 업무 번호 |
| Phase | 단계 |
| Vercel deploy / redeploy | 서버 배포 / 서버 재배포 |
| chat-log 인계 | 이전 작업 기록 / 인수인계 |
| 라우터 통합 | API 함수 통합 / 서버 함수 통합 |
| RPC / API | 서버 호출 / 서버 함수 |
| frontmatter | 메타데이터 / 머리말 |
| boundary 케이스 | 예외 상황 |
| fallback | 대체 동작 |
| null-safe | 빈 값 보호 |
| state: READY | 배포 완료 |
| HTTP 200 | 정상 응답 |
| commit hash | 작업 번호 |
| `dgmasters01/tw-b2b` 같은 repo 경로 | 코드 저장소 |

## 3. 사람용 탭 자동 추출기와의 약속

`admin-status.html`의 `loadHumanTab` 함수는 다음 규칙으로 동작한다:

1. **5블록 헤딩(`## 🎯`, `## 📍`, `## 🛠`, `## ✅`, `## ⏱`) 추출 시도**
2. **추출 성공** → 사업가 카드 UI로 표시 + 기술 상세는 `<details>` 접기
3. **추출 실패** (5블록 없음) → 자동 용어 변환(2번 표) 적용 후 표시 + ⚠️ 배지

**약속의 양방향성:**

- 5블록 누락 → 자동 변환 의존 → 변환 한계로 어색할 수 있음
- 5블록 정석 박기 → 카드 UI로 깔끔하게 표시

## 4. 강제 규칙 — 다시 묻지 않기

- 새 chat-log 박을 때마다 표준 적용 여부 대표님께 묻지 않는다 (이 표준이 답이다)
- 표준 위반 chat-log는 사람용 탭에서 ⚠️ 배지로 자동 표시 (Claude 자체 보고)
- 표준 변경은 헌법 6조 자가 검증 + 대표님 승인 필요
