## 1. 의무 1 — 첫 응답 5줄 양식 강제

모든 클로드 응답의 **첫 5줄**은 다음 양식 그대로:

```
[작업 소요: 약 X분 / N단계 / 변경 파일: ...]
🚦 ✅ 이 채팅 진행 가능 (또는 ⚠️ 새 채팅 권장)
📚 fetch 완료: boot.md / [작업파일] / 라이브 상태
🧭 북극성: [이 작업이 푸는 사용자 문제 한 줄]
🎯 한 채팅 한 결정: [본질 결정 한 줄]
🔍 중복 점검: grep 결과 한 줄 (없음 / N건 발견)
```

### 예시 (좋음)

```
[작업 소요: 약 25분 / 4단계 / 변경 파일: shared.css + admin-status.html]
🚦 ✅ 이 채팅 진행 가능 (토큰 여유 충분)
📚 fetch 완료: boot.md / shared.css / admin-status.html / tasks.json
🧭 북극성: 대표님이 admin 화면 낮 시간 볼 때 다크 강제로 눈부심 → 라이트 토글로 해소
🎯 한 채팅 한 결정: shared.css에 라이트 토큰 박기
🔍 중복 점검: shared.css에 [data-theme="light"] 이미 존재 → 추가 박기 금지
```

### 예시 (나쁨 — 헌법 위반)

```
안녕하세요 대표님! 작업 시작하겠습니다.    ← 5줄 양식 빠짐 = 위반
```

### 위반 시

대표님 **"헌법 확인"** 한 마디 → 즉시 응답 중단 + 자가 진단 + 재시작.

## 2. 의무 2 — 라이브 fetch + 중복 점검 강제

### 2-1. 라이브 fetch 의무

작업 시작 전 **인계서·tasks.json·chat-log만 믿지 말고, 작업 대상 파일/리소스를 GitHub에서 라이브로 fetch**해서 비교 보고.

**왜**: 인계서는 요약일 뿐이며 시간이 지나면 어긋난다. 진실원은 항상 GitHub 라이브.

**예시 사례 (2026-05-10 사고)**:
- 인계서: "BL-ADMIN-LIGHTMODE 33% / 2단계 완료 / 다음 시작 = 3단계 (shared.css에 라이트 박기)"
- 라이브: shared.css 이미 commit `afe0e9a`에서 라이트 블록 박혀있음
- 클로드가 라이브 fetch 안 했으면 → 중복 박기 사고

### 2-2. 중복 점검 grep 의무

새 코드/블록/토큰/함수/결정 박기 전 **반드시 동일 키워드 grep 후 결과 첫 응답 5번째 줄에 박음**.

```bash
# 예시: 부칙 16 박기 전
grep -rn "부칙 16\|BL-CLAUDE-DISCIPLINE" OPERATIONS_CHARTER.md _os/
```

grep 안 하고 박으면 = 임시 추가 사고 = 헌법 위반.

## 8. 끊김 객관 트리거 4종 (D-049, 2026-05-26 신설)

### 왜 신설했나
- 기존 부칙 16 "끊김 위험 감지 시 강제 권유" = **추상 규칙**
- Claude가 매번 "아직 괜찮겠지" 낙관 판단 → 끊김 → 사고 반복
- 2026-05-26 대표님 보고: "A로 자꾸 발생하네. 똑같은 상황이"
- **의지로 막는 규칙은 무조건 깨짐** → 객관 카운트 도입

### 4종 트리거 (판단 개입 0%, 카운트·감지만)

#### 트리거 1: 응답 횟수 (15회)
- Claude 응답 수가 15회를 넘기는 순간
- 그 다음 응답 첫 줄에 **무조건** 박음:
  > "🚨 끊김 위험 — 응답 N회 도달. 여기까지 commit 박고 새 채팅 권장"
- 작업 진행 중이라도 이 경고는 먼저 박은 후 작업 계속

#### 트리거 2: 파일 수정 횟수 (10회)
- `str_replace` + `create_file` 누적 호출이 10회를 넘기는 순간
- 동일 경고 박음
- 큰 파일·작은 변경 구분 없이 모두 1회로 셈 (보수적)

#### 트리거 3: 단계 완료 (commit 단위)
- `[step:done:N]` 태그 commit을 박을 때마다
- 그 commit 보고 직후 1줄 질문 박음:
  > "단계 N 완료. 끊고 새 채팅 갈까요, 계속할까요?"
- **헌법 1조 "자율 진행" 원칙과 충돌하지만 예외 인정** — 끊김 방지가 더 본질적
- 대표님 "계속"이라고 하면 다음 단계까지 자율 진행

#### 트리거 4: 거대 파일 감지 (1500줄+)
- 작업 대상 파일이 1500줄을 넘는다고 판단되면
- 그 작업 1개만 한 채팅에 박음 (다른 BL 같이 끼우지 말 것)
- 첫 응답에 명시:
  > "거대 파일 감지 ({파일명} N줄) — 이 채팅은 이 작업만"
