## 0. 왜 이 룰북이 필요한가

대표님 진단 (2026-05-15):

> "시스템 완성도를 사업을 서비스를 만들기 위한 방향이고, 실제적으로 서비스가 되었을때 서비스에 대한 모든걸 분석 판단하고 볼수 있는 곳은 admin.html 이곳에 있어야 되."

이전 사고 패턴:
- 클로드가 "사업 KPI 어디 박을까?" 망설임 → 시스템 완성도에 박으려 함 → 대표님 정정
- admin-status.html이 사업 KPI 안내한다는 가짜 링크(`/admin.html#business-kpi`) 박혀있는데 admin.html은 빈 로그인 페이지
- 결과: 데이터는 있는데 화면이 없음, 두 페이지 역할 모호

**해결**: 두 페이지를 **시간 축**으로 명확히 분리. 클로드가 매번 헷갈리지 않게 룰북에 박음.

---

## 1. 두 페이지 한 줄 정의

```
🛠 시스템 완성도 (admin-status.html) = "만드는 곳"
   사업을 만들기 위한 개발·수정·유지보수의 공간

💼 운영 대시보드 (admin.html)        = "굴리는 곳"
   서비스가 굴러갈 때 모든 걸 관리·분석·판단하는 공간
```

**시간 축으로 보면**:
```
[사업 만드는 동안]   ──→  [사업 굴러가는 동안]
  시스템 완성도              운영 대시보드
  (오늘~D-Day)              (D-Day~∞)

  ※ 사업 시작 후에도 시스템 완성도는 유지보수용으로 계속 살아있음
```

## 6. 미래 확장 — 시스템 매뉴얼 + 자동 유지보수

대표님 인사이트 (D-039 문맥):

> "추후 시스템이 다 많어지면, 어떤 기능이 있고 어떤식으로 시스템이 연동되었는지 너가 쉽게 파악할수 있도록 메뉴얼 같은것을 만들어서 제공해야 되. 그래야 추가 개발 및 에러 보완 같은것을 작업을 할수 있잖아."

이 인사이트의 구현 = `BL-SYSTEM-MANUAL-AUTOGEN`. 시스템 완성도 안에 다음 5섹션 자동 매뉴얼이 박힘:

1. **기능 인벤토리** — 어떤 기능들이 있나
2. **위치 지도** — 어느 페이지에 박혀있나
3. **데이터 연결도** — 어떤 테이블·API를 쓰나
4. **자동화 인벤토리** — 어떤 봇이 돌고 있나
5. **의존성 그래프** — 무엇과 무엇이 연결되어 있나

대표님 인사이트 (자동 유지보수):
> "추후 자동으로 유지보수가 되어야 된다. 이것도 추후 시스템을 만들어야 됨."

이 인사이트의 구현 = (미래 BL) `BL-AUTO-MAINTENANCE-SYSTEM`. 빨간불 → 자동 진단 → 자동 패치 → 자동 검증 → 자동 commit → 대표님 알림.

둘 다 시스템 완성도에 박히고, admin.html에는 안 박힘.

## 7. 헌법(OPERATIONS_CHARTER.md)에는 박지 않는 이유

부칙 14:
> "헌법 본문 200줄 이하 강제. 새 운영 룰은 헌법이 아닌 `_os/playbook/`에 박는다. charter-length-bot이 매 push 검사."

이 룰북은 **운영 룰**이지 메타 원칙이 아님. 헌법에는 큰 방향만 박힘 (부칙 5 = 4 시스템 카테고리). 페이지 역할 분리는 그 부칙 5의 디테일 구현 = playbook 위치.

**클로드 책임**: 매 채팅에서 "헌법에 박자"는 표현 금지. 정확한 자리는:
- 의사결정 → `DECISIONS.md` (D-XXX)
- 운영 룰 → `_os/playbook/*.md`
- 작업 → `tasks.json` (BL-XXX)
- 헌법 → 절대 부칙 추가 안 함 (정말 새 메타 원칙일 때만)
